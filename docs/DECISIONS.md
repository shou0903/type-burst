## D-001: Vertical Slice は BLITZ のみ

- **日付:** 2026-07-20
- **内容:** 実装順序(§30)に従い、30秒 BLITZ を先に完成させた。DUEL / PRIVATE ROOM / GHOST は未実装。
- **理由:** 仕様書 §0-4「まず30秒ソロ版を完成させる」。

## D-002: game-core の公開 API はクラスベース

- **内容:** §21 の `GameCore` インターフェース(純関数)ではなく、`BlitzGame` クラスを公開 API とした。内部は Seed + 呼び出し列に対して決定論的で、`Math.random()` は使用しない。
- **理由:** ソロ版では権威サーバーとの状態共有が不要で、クラスの方がイベント発行と統合しやすい。Phase 5(Online Duel)で純関数 API へ再編する。盤面操作(`findGroup` / `applyGravity` / `findAutoGroups`)は純関数として実装済みで、再編時にそのまま流用できる。

## D-003: Tier のモーラ帯域を実データに合わせて調整

- **内容:** short 4〜7 / standard 8〜12 / long 13〜26 モーラとした(仕様書 §7.2 は short 5〜9 / standard 10〜17 / long 18〜26)。
- **理由:** 仕様書付属のサンプルデータ(20件)の tier 付けが §7.2 の帯域と一致しない(例: 「電車が駅に到着した」=13モーラが long)。サンプルの意図を優先し、全データが検証を通る帯域に再定義した。`moraCount` は手書きせず `countMora()` で読み仮名から自動計算する。

## D-004: BLITZ のトップアウトは即時終了

- **内容:** 行上昇でブロックが盤面最上段(10行)を超えたら、その時点でゲーム終了(OVERFLOW 表示)。スコアは有効。
- **理由:** 仕様書は BLITZ のトップアウトを定義していない(§5.2 の危険ライン敗北は DUEL の条件)。30秒の緊張感を保ちつつ、稼いだスコアを無効にしない。

## D-005: KPM の分母は実プレイ経過時間

- **内容:** `KPM = correctKeyCount / 実プレイ秒数 × 60`。連鎖演出中もプレイ時間に含める。
- **理由:** §18 の `activeTypingSeconds` の定義が曖昧なため、寿司打等の慣例に合わせて全プレイ時間とした。将来、盤面選択停止時間を別指標として記録する。

## D-006: 入力ロックは全ての消去解決中に適用

- **内容:** 連鎖中だけでなく、単独消去の解決中(ヒットストップ〜落下、約340ms)も新規入力を受け付けない。
- **理由:** §10.5「連鎖中は新しい入力を受け付けない」を消去解決全体に一般化。落下中の盤面へ入力すると候補選択が不安定になるため。

## D-007: 描画は Canvas 2D 直描画

- **内容:** Phaser / PixiJS を使わず、Canvas 2D API で直接描画する。
- **理由:** §21 は「PhaserまたはPixiJS相当の2D描画」。6×10ブロック+パーティクル程度なら Canvas 2D で60fpsを維持でき、依存とバンドルサイズを大幅に削減できる。ゲームロジックは React コンポーネントにも Canvas にも置かず `game-core` に隔離済み。

## D-008: 効果音は Web Audio シンセ生成

- **内容:** 音声ファイルを使わず、オシレータとノイズバッファで効果音を合成する。
- **理由:** アセット制作前にゲームフィールを検証する(Starter Prompt「初期段階では仮の四角形と簡単な効果音で構いません」)。連鎖ごとの音程上昇(§15)はスケール配列で実装。

## D-009: 長音「ー」は「-」キーで入力

- **内容:** 読み仮名に含まれる「ー」(ノート、ペース等)はハイフンキー1打で入力する。
- **理由:** サンプルデータに「のーと」「ぺーす」が含まれる。一般的なローマ字入力の慣例に従う。

## D-010: 妨害ブロックはデータ・ロジックのみ実装

- **内容:** 妨害文章(8件)、無属性ブロックの隣接破壊、攻撃力計算式は game-core に実装済みだが、BLITZ では妨害ブロックが出現しない。
- **理由:** §12-13 は DUEL の要素。Phase 4(Ghost)で疑似妨害として利用する。

## D-011: 結果画面の「苦手傾向」はヒューリスティック

- **内容:** 苦手かな・ローマ字遷移の統計(§18)は未実装。結果画面には正確率・連鎖数などから選ぶ定型アドバイス1件を表示する。
- **理由:** Vertical Slice の範囲外。Phase 6 でキー単位のミス統計を実装する。

## D-012: BLITZ を廃止しサバイバル制へ変更(v2)

- **日付:** 2026-07-20
- **内容:** 「30秒は短すぎる。時間を終了条件にしないでほしい」というプレイテストのフィードバックを受け、時間切れを廃止。行上昇が経過時間とともに加速し(5.5秒→最短2.4秒)、盤面があふれるまで続くサバイバル制にした。仕様書 §5.1 の BLITZ(30秒)とは異なる。
- **理由:** ユーザーフィードバック最優先。KPI「もう一戦率」はサバイバルの生存時間更新動機で担保する。

## D-013: 特殊ブロックと TYPE BURST の追加(v2)

- **内容:** 仕様書 §35 は「スキル」をMVP外としているが、フィードバック「必殺技や特殊ブロックが欲しい」を受けて以下を追加した。
  - **ボム**(出現率7%): 入力完了で周囲3×3を爆破
  - **プリズム**(出現率3.5%): 入力完了で盤面最多属性を全消し
  - **TYPE BURST**: 消去と連鎖でゲージが溜まり、満タン時に Enter で下3行を吹き飛ばす。バースト起因の消去ではゲージは再充填されない(無限ループ防止)
- **理由:** キャラクター・RPG要素ではなく盤面ギミックの範囲に留め、§0-9「爆発・落下・連鎖の気持ちよさ優先」に沿う形で採用。

## D-014: CPU対戦(DUEL の先行実装)

- **内容:** オンライン対戦(Phase 5)の前に、同一 game-core を使う CPU 対戦を実装した。CPU は結果を書き換えず実際にキーを1打ずつ入力する(kpm・ミス率・思考時間は難易度3段階の Config)。妨害送信・相殺・着弾(§12-13)はこのモードで稼働。時間制限はなく、相手の盤面をあふれさせたら勝ち。
- **理由:** 初期過疎対策(§5.4 GHOST と同目的)+ 対戦の面白さを早期検証。CPU も Seeded PRNG 駆動なので試合全体が決定論的でリプレイ可能。

## D-015: 対戦の勝敗判定は トップアウトのみ

- **内容:** §5.2 の「危険ライン超え2秒維持で勝利」は未実装。盤面最上段を超えた側の負けのみ。
- **理由:** 時間制限を廃止したため時間切れ判定が不要になり、ルールを1つに絞った方が分かりやすい。2秒ルールはオンライン版で再検討する。

## D-016: 行のせり上がりを「上からの落下」へ変更(v3)

- **日付:** 2026-07-20
- **内容:** 仕様書 §11 の「下から1行追加し全体を押し上げる」を廃止し、新しい行は盤面の上から降ってきて各列の山の一番上に積まれる方式にした。既存ブロックは一切動かない。
- **理由:** フィードバック「上から降ってくるほうが分かりやすいし演出的にもいい」。副次効果として、入力中のブロックが行追加で動かなくなり、タイピングの継続性も向上した。

## D-017: 選択キャンセル操作の追加(v3)

- **内容:** Esc または Backspace で入力中・ロック中の選択を解除できる。キャンセルはミスに数えず、正確率にも影響しない。
- **理由:** フィードバック「入力を開始したブロックをキャンセルしたい」。§9.3「バックスペースを必須にしない」とは矛盾しない(必須ではなく任意の救済手段)。

## D-018: ALL CLEAR ボーナス・LEVEL・ランク評価(v3)

- **内容:** 楽しさを増す進行システムを追加した。
  - **ALL CLEAR**: 盤面を空にすると +5000点・ゲージ+50。次の行が600ms以内に降ってくる
  - **LEVEL**(サバイバル): 30秒生存ごとにレベルアップ(+レベル×300点)。行上昇の加速と合わせて進行感を可視化
  - **ランク評価**(サバイバル結果): スコアで S+/S/A/B/C/D を表示
  - HUD に PERFECT ストリーク(🔥×n)を表示
- **理由:** フィードバック「もっとゲームが楽しくなる機能を」。すべて盤面ルールを変えない加点・表示系で、§36 の優先順位(連鎖の気持ちよさ)を損なわない。

## D-019: 特殊ブロックのバランス調整(v4)

- **日付:** 2026-07-20
- **内容:** フィードバック「行上昇が速くなると通常ブロックをほとんど打たず、ボム/プリズムしか打たなくなり、同じ文章を繰り返し打つことになって練習にならない」を受け、2点を修正した。
  1. ボム・プリズムの読み仮名を強制的に short tier(全10件)から選ぶのをやめ、通常ブロックと同じ Tier 分布(short 20% / standard 65% / long 15%、全53件)から選ぶように変更
  2. 盤面全体でボム・プリズムをそれぞれ**同時に最大1個まで**に制限(1行1個の制約は維持)。既存の抽選ロールは、上限に達している種類の分を消費するだけで通常ブロックへフォールバックし、もう片方の出現率を誤って底上げしないようにした
- **理由:** 短文プール(10件)だけを繰り返し引くため文章が単調になり、かつ「短い労力で盤面を大きく消せる」効率の良さから上級者が通常ブロックを無視して特殊ブロックだけを狩る戦略が最適化されてしまっていた。盤面全体で2個(ボム1+プリズム1)までに絞ることで、残り4列以上は通常ブロックの連鎖でしか対処できず、タイピング練習としての多様性が強制的に確保される。

## D-020: ビューポート縮小時のレイアウト保護(v4)

- **内容:** Chromeのブックマークバー表示など、実効ビューポート高さが減った場合に画面上部が到達不能になっていた問題を修正した。原因は `.screen` が `height:100%` 固定+中央寄せで、`body{overflow:hidden}` と組み合わさっていたため、コンテンツがビューポートより高いと中央寄せのはみ出しが上下均等に発生し、上方向のはみ出し分がドキュメント座標マイナスとなってスクロールしても永久に見えなくなっていたこと。
  - `body` を `overflow-y: auto` に変更(下方向のはみ出しは常にスクロールで到達可能にする保険)
  - `.screen` を `justify-content: flex-start` に変更(はみ出しを常に下方向のみへ逃がし、上部が埋もれる経路を構造的に排除)
  - `useFitToViewport` フック(`apps/web/src/hooks/useFitToViewport.ts`)を追加。コンテンツがビューポートに収まらない場合、`transform: scale()` で自動的に縮小する(最小スケール0.55、それ以上縮めても読みにくくなるため下限を設け、それでも収まらない極小ウィンドウはスクロール併用にフォールバック)。ResizeObserver(コンテンツ自身+documentElementの双方を監視)・window resizeイベント・500msポーリングの3系統で検知し、いずれか1つが効かない環境でも追従する
- **理由:** ブックマークバーの表示/非表示はページのコンテンツサイズを一切変えずビューポートだけを変えるため、コンテンツ側だけを監視していると検知できない。検証で使用したブラウザツールの `resize_window` はCDP経由のビューポート上書きで、実ブラウザと異なり `resize` イベントも `ResizeObserver` も発火しないことが判明したため、ポーリングを保険として追加した(実ブラウザでは通常イベントベースの検知だけで十分機能する)。

## D-023: アクセシビリティ拡充(High Contrast・文字サイズ・OS設定反映)

- **内容:** 仕様書§20の残項目を実装した。

## D-024: ブラウザ互換性の検証範囲

- **内容:** この開発環境で直接操作できるブラウザは Chromium 系(Chrome/Edge相当)のみ。Firefox・Safari の実機検証は行っていない。以下は API サポート状況に基づく書面上の分析。
  - ResizeObserver: Safari 13.1+ / Firefox 69+ で対応済み(問題なし)
  - Web Audio API: 全ブラウザ対応。Safari は自動再生ポリシーが厳しいが、本実装は `AudioContext` をユーザーのクリック操作(サバイバル/対戦開始ボタン)後に初期化しており準拠済み
  - `structuredClone`(対戦成績の保存に使用): Safari 15.4+ / Firefox 94+ / Chrome 98+。仕様書が対象とする「現行安定版」であれば全て対応範囲内
  - CSS カスタムプロパティ(`--font-scale`)・`transform: scale()`: 全モダンブラウザ対応
  - `KeyboardEvent.isComposing`(IME検出): 全モダンブラウザ対応
- **理由:** 実機を用意できない制約下で、根拠のない「動作確認済み」を主張しないため、検証済み範囲(Chromium系・本セッションで多数回実施)と未検証範囲(Firefox・Safari実機)を明確に分けて記録する。
- **推奨:** 公開前に Firefox・Safari(可能なら実iPad/Mac含む)での簡易な手動確認を1回行うこと。特に IME 検出まわりの挙動と、Safari の音声再生解禁タイミングを重点確認。

## D-025: ホスティングをCloudflare PagesからVercelへ変更

- **日付:** 2026-07-20
- **内容:** 当初推奨した Cloudflare Pages(D-024以前の推奨)は、npm workspacesモノレポとの組み合わせで3段階連続の障害(wranglerのワークスペース検出エラー→APIトークン権限不足→ビルドトークン自体の内部不具合)に遭遇し、最後の問題はCloudflare側のUIが提示する修正手順を試しても解消しなかった。原因究明を打ち切り、Vercelへ切り替えたところ即座に成功した。
- **理由:** ローカルビルドは常に成功しており、こちら側の設定・コードに問題がないことは検証済みだった。Cloudflareの新しい「Workers Builds」統合パイプラインがモノレポ構成に対して現状脆弱と判断し、時間を浪費するより実績のあるVercelへ切り替える判断をした。
- **現在の本番URL:** https://type-burst-web.vercel.app/(詳細は docs/DEPLOYMENT.md)
- **今後:** Cloudflareの帯域無制限という利点を活かしたい場合は、`docs/DEPLOYMENT.md` の「経緯」セクションを参照のうえ再挑戦する。`wrangler.toml` はリポジトリに残置しているが現在未使用。

## D-026: 世界ランキング機能の追加(初のバックエンド導入)

- **理由:** 「登録なし」の方針(仕様書§4等)は維持しつつ、ランキング参加だけは任意入力にすることで両立させた。

## D-027: 広告表示の追加(仕様書からの方針転換)

- **日付:** 2026-07-20
- **内容:** 仕様書§28は将来の収益化でも「広告なし」「Pay to Win禁止」と明記していたが、ユーザー要望により画面両サイドへのGoogle AdSense広告表示を追加した。これは明確な方針転換であり、ここに記録する。
  - `apps/web/src/adsConfig.ts` にプレースホルダーのpublisher ID/スロットIDを配置。実際のAdSense審査通過後にこのファイルのみ書き換えれば有効化される
  - 未設定(プレースホルダーのまま)の間は広告を一切読み込まない安全側の実装(`isAdsenseConfigured()`)
  - GDPR等への配慮として同意バナー(`ConsentBanner.tsx`)を実装し、同意("accepted")を得るまでAdSenseスクリプト自体を読み込まない
  - 広告はビューポート幅1450px以上でのみ表示(ゲーム盤面との重なりを避けるため)。ゲームの進行・結果・スコアには一切影響しない(Pay to Winは行わない、利用規約に明記)

## D-031: CPU対戦中に盤面サイズが変わったり戻ったりする不具合の修正

- **日付:** 2026-07-21
- **内容:** CPU対戦モードで、妨害の送受信・PERFECTコンボ・DANGER警告などのタイミングで画面全体(盤面+HUD)の縮尺が繰り返し変化し、プレイしづらいという報告への対応。
  - **原因:** `useFitToViewport`(画面がビューポートより大きい場合に縮小するフック)はコンテンツの自然な高さ(`scrollHeight`)を`ResizeObserver`で監視し、変化するたびに再計算・再スケールする。`GameScreen.tsx`内の`.streak-badge`(PERFECT×N)・`.cpu-incoming`(CPUへの妨害数▼N)・`.incoming-badge`(妨害接近▼N)・`.danger-badge`(DANGER!)の4つは、対戦中の状態変化に応じて条件付きでDOMへ着脱される実装になっていた。これらが出現/消失するたびに`.screen.game`コンテナの実寸高さが変わり、`useFitToViewport`が反応して盤面全体を拡大縮小していた。
  - 同種の問題は`.hud-typing`(入力欄)については既に`min-height: 118px`で予防済みだったが、後から追加されたこれら4つのバッジには同じ対策が漏れていた。
  - **修正:** 4つのバッジをDOMへの着脱ではなく常時マウントしたうえで、非表示時は`visibility: hidden`(`.badge-reserved.badge-hidden`)にする方式へ変更(`apps/web/src/screens/GameScreen.tsx`, `apps/web/src/styles.css`)。`visibility: hidden`はレイアウト上の領域を保持したまま見た目だけ隠すため、表示/非表示に関わらずコンテナの高さが変化しなくなる。
  - ブラウザで`player.danger`等を直接書き換えて検証し、バッジのON/OFFに関わらず`el.scrollHeight`と`transform: scale(...)`が完全に一定であることを確認済み。
- **理由:** 画面が動いてしまう体感の根本原因は「バッジの着脱によるレイアウト変化」であり、`useFitToViewport`側の再計算ロジック自体は(ブックマークバーの表示切替など)正しく必要な機能だったため、原因側(バッジの実装)を直す方針とした。

## D-033: サバイバル難易度の実現方法を「行上昇の速さ」から「文章の長さ」へ変更

- **日付:** 2026-07-21
- **内容:** D-032で実装した難易度(easy/normal/hard)は行上昇の速さ(RiseConfig)を変える方式だったが、本番反映後にユーザーから「難易度を下げれば行上昇が十分遅くなり、積み上がらないまま無限に生存してスコアを無限に稼げてしまう」という指摘を受けた。
  - **対応:** 行上昇の速さ(`config.survivalRise`)を難易度に関わらず完全に共通の1種類に戻し、代わりに1ブロックあたりに要求される文章の長さの配分(`tierRatio`: short/standard/long)を難易度ごとに変える方式に変更した。
    - easy: `{ short: 0.6, standard: 0.35, long: 0.05 }`(短い文章が多く、素早くさばける)
    - normal: `{ short: 0.2, standard: 0.65, long: 0.15 }`(既存のデフォルトのまま)
    - hard: `{ short: 0.05, standard: 0.45, long: 0.5 }`(長い文章が多く、要求打鍵数が多い)
  - `packages/game-core/src/survival.ts`: `PlayerCore`へ渡す`config`を`{ ...config, tierRatio: profile.tierRatio }`で複製し、`rise`は常に`config.survivalRise`(共通)を渡すよう変更。CPU対戦(`DuelGame`)の`tierRatio`には影響しない。
  - スコア倍率(D-032で導入)は維持。行上昇ペースという「終了までの上限」が全難易度共通になったことで、無限にスコアを稼げる問題自体は構造的に解消された。
- **理由:** 「盤面が積み上がるペース」という難易度の根幹部分を難易度間で変えてしまうと、緩い設定側の上限が事実上なくなってしまう。文章の長さで難易度を調整すれば、上限(行上昇ペース)は共通のまま「同じ時間内にどれだけ多くのブロックをさばけるか」という体感難易度だけを変えられる。
- **教訓:** 難易度調整は「終了条件に影響するパラメータ」と「プレイの快適さに影響するパラメータ」を分けて考える必要がある。前者を難易度間で変えると、ゲームの根本的なバランス(このゲームでは「無限には生存できない」という前提)が崩れる。

## D-038: サバイバル難易度の再修正(行上昇の速さを難易度ごとに復活)

- **日付:** 2026-07-21
- **内容:** D-033で「行上昇の速さは全難易度共通、文章の長さ(tierRatio)だけで難易度を付ける」方式に変更したが、本番反映後にユーザーから「難易度の調整が全然できていない。初級の単語が長すぎて初心者は全然時間ない」との指摘を受けた。
  - **原因分析:** 文章の長さだけを短くしても、そもそも1行あたりに与えられる時間(行上昇の間隔)自体がnormalと同じままでは、初心者がタイピングに慣れるための猶予が根本的に不足する。実際のフレーズデータ(`packages/phrase-content`)を確認したところ、short tierは全61件中10件のみで、そのshort tierですらローマ字換算で8〜12文字程度あり、文章を短くする調整だけでは初心者の「読んで打ち始めるまでの時間不足」を解消しきれないことを確認した。
  - **対応:** `packages/game-core/src/config.ts`の`SurvivalDifficultyProfile`に`rise: RiseConfig`を復活させ、行上昇の速さを再び難易度ごとに変える。ただし、D-032で問題になった「easyの行上昇が遅すぎて無限に生存できる」という状態を再発させないよう、以下の設計にした。
    - **開始間隔(`startIntervalMs`)と加速度(`accelPerSecondMs`)は難易度間で大きく差をつける**(easy: 9500ms開始・10ms/秒、normal: 5500ms開始・25ms/秒、hard: 4200ms開始・32ms/秒)。序盤の猶予が初心者の生存体験を大きく左右するため、ここを主な調整レバーにした。
    - **最終的な最短間隔(`minIntervalMs`、floor)はeasyとnormalで差を小さく保つ**(easy: 2800ms、normal: 2400ms、hard: 1700ms)。D-032のeasy(4000ms)はnormal(2400ms)との差が大きすぎて無限生存の温床になっていたため、今回は差を400msに抑え、プレイし続ければ最終的にnormal相当の圧力へ収束するようにした。
    - `tierRatio`によるD-033の調整(easyは短い文章が多い)は維持しつつ、easyをわずかにさらに短文寄りに調整(`{ short: 0.6, standard: 0.37, long: 0.03 }`、旧`{0.6, 0.35, 0.05}`)。

- **理由:** 難易度調整には「終了条件の上限(floor)」と「序盤の体感的な猶予(start/accel)」という2つの異なるパラメータがあり、D-033はこれらを区別せず両方を難易度間で共通化してしまったため、初心者救済という本来の目的を達成できていなかった。floorの差を小さく保つことでD-032の無限生存問題を再発させずに、序盤の猶予だけを難易度ごとに大きく変えることで両立を狙う。
- **教訓:** 「無限に有利になる」問題と「序盤に十分な猶予を与える」要求は本質的に別物であり、片方を潰すために両方を犠牲にする必要はない。パラメータを分解し、どの部分が終了条件(上限)に効き、どの部分が体感難易度(序盤の余裕)に効くかを切り分けて設計すべきだった。

## D-041: 初級の「無限に生存できる」問題を修正し、ランキングを難易度別に分離

- **日付:** 2026-07-21
- **内容:** D-040でeasyをmicro tier(単語レベル)中心にしたところ、ユーザーから「初級だと一生できてしまう。そこら辺のゲームバランスの調整をもっと考えてしてほしい」「ランキングはやっぱり難易度別に分けたほうがいい」との指摘を受けた。これは2つの独立した修正として対応した。
  - **①ゲームバランス(無限生存の修正):**
    - **原因分析:** 行上昇の間隔には下限(`minIntervalMs`、floor)があり、D-040時点では2400msだった。floorに達すると行上昇はそれ以上速くならないため、1行(6ブロック)をfloor間隔内に安定してさばき続けられる難易度は理論上無限に生存できてしまう。easyのmicro tier(2〜4モーラの単語)は非常に速く打てるため、2400ms/6ブロック=平均400ms/ブロックというfloorのペースを上回るのが可能になり、実際に「一生できる」状態が生まれていた。normal/hardは文章が長く、floorのペースにいずれ追いつかれるため問題化していなかった。
    - **対応:** `packages/game-core/src/config.ts`の共通`survivalRise`のfloorを2400ms→1200msへ引き下げ、加速度も25→30ms/秒に強化した。**floorは全難易度共通のパラメータであり、これを調整してもユーザーが指定した「難易度間で行上昇速度を変えない」制約には抵触しない**(難易度差はtierRatioのみで維持)。これにより、micro中心のeasyであっても長時間プレイすれば必ず盤面が積み上がり終了するようにした。
    - この調整は実プレイデータに基づく厳密な検証ではなく、理論的な妥当性(1行あたりの平均処理時間の見積もり)に基づく暫定値。実際のプレイで依然として無限生存に近い状態が起きる場合は、floorのさらなる引き下げや、片手落ちにならないよう他の指標(例: 選択ブロックの視認・ロック時間)も含めた見直しが必要になる可能性がある。
  - **②ランキングの難易度別分離:**
    - **対応:** これまでは1本の「全難易度共通ランキング」に対し`scoreMultiplier`(D-032由来のスコア補正係数)を掛けて公平性を保つ設計だったが、難易度ごとに完全に別のランキングへ分離する方式に変更した。
      - `apps/web/api/scores.ts`: Redisのsorted setキーを`leaderboard:survival:alltime`の1本から`leaderboard:survival:alltime:{easy|normal|hard}`の3本に分離。GETエンドポイントは`?difficulty=`クエリパラメータを受け取り(未指定時はnormal)、該当難易度のランキングのみ返す。POST時のスコア補正(`SCORE_MULTIPLIER`)は不要になったため撤去し、素点をそのままランキングに使用する。
      - `packages/game-core/src/config.ts`・`survival.ts`・`types.ts`: 難易度別ランキングにより不要になった`scoreMultiplier`(`SurvivalDifficultyProfile`・`SurvivalSummary`)を削除。
      - `apps/web/src/ranking.ts`: `fetchTopScores(difficulty, limit)`に難易度引数を追加。`RankingEntry`から`rawScore`(補正前後の区別)を削除しシンプルな`score`のみに。
      - `apps/web/src/screens/RankingScreen.tsx`: ランキング画面に初級/中級/上級のタブ(既存の`chip`/`chip-active`スタイルを流用)を追加し、選択した難易度のランキングのみ表示。テーブルの「難易度」列(旧`ranking-difficulty-chip`)は不要になったため削除。
    - 旧・共通ランキングのRedisデータ(`leaderboard:survival:alltime`キー)は削除せずそのまま残し、新しい難易度別キーへの移行は行わない(過去データは表示されなくなるが、破壊的操作を避けるため放置)。
- **理由:** ①は「常に共通の行上昇ペースがある以上、いずれ誰でも敗北する」という設計の大前提が、tierRatio側の調整幅が広がったことで崩れていたための構造的な修正。②は「難易度が違えば比較対象として意味がない」というユーザーの直感的な指摘であり、スコア補正による正規化よりも単純に分離した方が納得感が高い。
- **教訓:** 「行上昇速度は共通」という制約と「floor(共通パラメータ)の値」は別物であり、難易度ごとの体験(tierRatio)を大きく変えるたびに、共通パラメータ側がその変化に対して依然として妥当かを再検証する必要がある。また、スコアの公平性は「補正して1本化する」よりも「土俵を分ける」方が、ユーザーにとって直感的で説明不要な場合が多い。
