# PERFORMANCE BUDGET

仕様書 §2.2・§33 に基づく性能予算。

## 目標値

| 項目 | 予算 |
|------|------|
| フレームレート | 通常時 60fps(最低 55fps) |
| 入力→画面反応 | 50ms 未満 |
| ページ表示→プレイ開始 | 20秒以内(実測は数秒) |
| 初回バンドル(gzip) | 150KB 以下 |
| メモリ | 30分連続プレイで増加が横ばい |

## 現在の実装での対策

- **描画:** Canvas 2D 一枚。React の再レンダリングは HUD のみ(盤面は触らない)
- **パーティクル:** 1爆発 14個、Reduced Motion 時 4個。寿命で自動回収
- **フォント計測:** 文章ごとの fit サイズを Map でキャッシュ(measureText は初回のみ)
- **dt クランプ:** タブ復帰時の巨大 delta を 100ms に制限し、スパイラルを防ぐ
- **Audio:** ノードは短寿命(stop 後 GC)。AudioContext は1個を使い回す
- **連鎖上限:** 1解決あたり最大32段(無限ループ防止、§32)

## 計測方法

- Chrome DevTools Performance で 10 CHAIN 発生時のフレーム時間を確認
- `npm run build` 後の dist サイズを確認
- 長時間プレイは DevTools Memory のヒープスナップショット比較

## 退行チェックリスト(機能追加時)

1. 60ブロック + 大量パーティクル + シェイクの同時発生で 60fps を維持しているか
2. keydown → feedKey → 次フレーム描画の経路に同期の重い処理を入れていないか
3. 毎フレームのアロケーションを増やしていないか(Snapshot 生成は許容済み)

## 実測結果(2026-07-20、開発機での直接計測)

CPUスロットリング機能を持つ計測ツールが手元になかったため、最も重い処理の実行時間を直接計測し、
60fps予算(16.6ms/フレーム)に対する余裕度から低スペック環境での挙動を推定した。

| 計測対象 | 結果 | 備考 |
|---|---|---|
| `game-core`: 全消去+連鎖1回分の `advance()` | < 0.1ms(計測不能なほど高速) | 純ロジックはボトルネックにならない |
| `feedKey()` 1文字あたり | 0.6ms(22文字の文章1本分の合計) | |
| Canvas描画1フレーム(8連鎖+BURST発火を単一フレームに集約した最悪ケース) | 初回 11.8ms、以降 0.6〜2.2ms | 予算16.6msに対し初回でも4.8ms余裕。実際のゲームでは連鎖は150ms間隔で段階的に発生するため、パーティクル生成はフレームをまたいで分散し、この計測はさらに悲観的な想定になっている |

**結論:** 現行実装は開発機で大幅な余裕(初回スパイクでも約1.4倍の余裕)があり、2〜3倍遅い環境でも
単発フレームの一時的な落ち込みに留まる見込み。ただし実機の低スペックPCでの体感確認はリリース前に
一度行うことを推奨する(Chrome DevTools Performance タブの「CPU: 4x slowdown」等で追試可能)。
