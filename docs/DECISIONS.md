# DECISIONS

仕様書(TYPE_BLAST_Product_Game_Technical_Spec_v1.0.md)からの決定・変更・補完の記録。
新しい決定は末尾に追記する。

---

## D-001: Vertical Slice は BLITZ のみ

- **日付:** 2026-07-20
- **内容:** 実装順序(§30)に従い、30秒 BLITZ を先に完成させた。DUEL / PRIVATE ROOM / GHOST は未実装。
- **理由:** 仕様書 §0-4「まず30秒ソロ版を完成させる」。

## D-002: game-core の公開 API はクラスベース

- **内容:** §21 の `GameCore` インターフェース(純関数)ではなく、`BlitzGame` クラスを公開 API とした。内部は Seed + 呼び出し列に対して決定論的で、`Math.random()` は使用しない。
- **理由:** ソロ版では権威サーバーとの状態共有が不要で、クラスの方がイベント発行と統合しやすい。Phase 5(Online Duel)で純関数 API へ再編する。盤面操作(`findGroup` / `applyGravity` / `findAutoGroups`)は純関数として実装済みで、再編時にそのまま流用できる。

## D-003: Tier のモーラ帯域を実データに合わせて調整

- **内容:** short 4〜7 / standard 8〜12 / long 13〜26 モーラとした(仕様書 §7.2 は short 5〜9 / standard 10〜17 / long 18〜26)。
- **理由:** 仕様書付属のサンプルデータ(20件)の tier 付けが §7.2 の帯域と一致しない(例: 「電車が駅に到着した」=13モーラが long)。サンプルの意図を優先し、全データが検証を通る帯域に再定義した。`moraCount` は手書きせず `countMora()` で読み仮名から自動計算する。

## D-004: BLITZ のトップアウトは即時終了

- **内容:** 行上昇でブロックが盤面最上段(10行)を超えたら、その時点でゲーム終了(OVERFLOW 表示)。スコアは有効。
- **理由:** 仕様書は BLITZ のトップアウトを定義していない(§5.2 の危険ライン敗北は DUEL の条件)。30秒の緊張感を保ちつつ、稼いだスコアを無効にしない。

## D-005: KPM の分母は実プレイ経過時間

- **内容:** `KPM = correctKeyCount / 実プレイ秒数 × 60`。連鎖演出中もプレイ時間に含める。
- **理由:** §18 の `activeTypingSeconds` の定義が曖昧なため、寿司打等の慣例に合わせて全プレイ時間とした。将来、盤面選択停止時間を別指標として記録する。

## D-006: 入力ロックは全ての消去解決中に適用

- **内容:** 連鎖中だけでなく、単独消去の解決中(ヒットストップ〜落下、約340ms)も新規入力を受け付けない。
- **理由:** §10.5「連鎖中は新しい入力を受け付けない」を消去解決全体に一般化。落下中の盤面へ入力すると候補選択が不安定になるため。

## D-007: 描画は Canvas 2D 直描画

- **内容:** Phaser / PixiJS を使わず、Canvas 2D API で直接描画する。
- **理由:** §21 は「PhaserまたはPixiJS相当の2D描画」。6×10ブロック+パーティクル程度なら Canvas 2D で60fpsを維持でき、依存とバンドルサイズを大幅に削減できる。ゲームロジックは React コンポーネントにも Canvas にも置かず `game-core` に隔離済み。

## D-008: 効果音は Web Audio シンセ生成

- **内容:** 音声ファイルを使わず、オシレータとノイズバッファで効果音を合成する。
- **理由:** アセット制作前にゲームフィールを検証する(Starter Prompt「初期段階では仮の四角形と簡単な効果音で構いません」)。連鎖ごとの音程上昇(§15)はスケール配列で実装。

## D-009: 長音「ー」は「-」キーで入力

- **内容:** 読み仮名に含まれる「ー」(ノート、ペース等)はハイフンキー1打で入力する。
- **理由:** サンプルデータに「のーと」「ぺーす」が含まれる。一般的なローマ字入力の慣例に従う。

## D-010: 妨害ブロックはデータ・ロジックのみ実装

- **内容:** 妨害文章(8件)、無属性ブロックの隣接破壊、攻撃力計算式は game-core に実装済みだが、BLITZ では妨害ブロックが出現しない。
- **理由:** §12-13 は DUEL の要素。Phase 4(Ghost)で疑似妨害として利用する。

## D-011: 結果画面の「苦手傾向」はヒューリスティック

- **内容:** 苦手かな・ローマ字遷移の統計(§18)は未実装。結果画面には正確率・連鎖数などから選ぶ定型アドバイス1件を表示する。
- **理由:** Vertical Slice の範囲外。Phase 6 でキー単位のミス統計を実装する。

## D-012: BLITZ を廃止しサバイバル制へ変更(v2)

- **日付:** 2026-07-20
- **内容:** 「30秒は短すぎる。時間を終了条件にしないでほしい」というプレイテストのフィードバックを受け、時間切れを廃止。行上昇が経過時間とともに加速し(5.5秒→最短2.4秒)、盤面があふれるまで続くサバイバル制にした。仕様書 §5.1 の BLITZ(30秒)とは異なる。
- **理由:** ユーザーフィードバック最優先。KPI「もう一戦率」はサバイバルの生存時間更新動機で担保する。

## D-013: 特殊ブロックと TYPE BURST の追加(v2)

- **内容:** 仕様書 §35 は「スキル」をMVP外としているが、フィードバック「必殺技や特殊ブロックが欲しい」を受けて以下を追加した。
  - **ボム**(出現率7%): 入力完了で周囲3×3を爆破
  - **プリズム**(出現率3.5%): 入力完了で盤面最多属性を全消し
  - **TYPE BURST**: 消去と連鎖でゲージが溜まり、満タン時に Enter で下3行を吹き飛ばす。バースト起因の消去ではゲージは再充填されない(無限ループ防止)
- **理由:** キャラクター・RPG要素ではなく盤面ギミックの範囲に留め、§0-9「爆発・落下・連鎖の気持ちよさ優先」に沿う形で採用。

## D-014: CPU対戦(DUEL の先行実装)

- **内容:** オンライン対戦(Phase 5)の前に、同一 game-core を使う CPU 対戦を実装した。CPU は結果を書き換えず実際にキーを1打ずつ入力する(kpm・ミス率・思考時間は難易度3段階の Config)。妨害送信・相殺・着弾(§12-13)はこのモードで稼働。時間制限はなく、相手の盤面をあふれさせたら勝ち。
- **理由:** 初期過疎対策(§5.4 GHOST と同目的)+ 対戦の面白さを早期検証。CPU も Seeded PRNG 駆動なので試合全体が決定論的でリプレイ可能。

## D-015: 対戦の勝敗判定は トップアウトのみ

- **内容:** §5.2 の「危険ライン超え2秒維持で勝利」は未実装。盤面最上段を超えた側の負けのみ。
- **理由:** 時間制限を廃止したため時間切れ判定が不要になり、ルールを1つに絞った方が分かりやすい。2秒ルールはオンライン版で再検討する。

## D-016: 行のせり上がりを「上からの落下」へ変更(v3)

- **日付:** 2026-07-20
- **内容:** 仕様書 §11 の「下から1行追加し全体を押し上げる」を廃止し、新しい行は盤面の上から降ってきて各列の山の一番上に積まれる方式にした。既存ブロックは一切動かない。
- **理由:** フィードバック「上から降ってくるほうが分かりやすいし演出的にもいい」。副次効果として、入力中のブロックが行追加で動かなくなり、タイピングの継続性も向上した。

## D-017: 選択キャンセル操作の追加(v3)

- **内容:** Esc または Backspace で入力中・ロック中の選択を解除できる。キャンセルはミスに数えず、正確率にも影響しない。
- **理由:** フィードバック「入力を開始したブロックをキャンセルしたい」。§9.3「バックスペースを必須にしない」とは矛盾しない(必須ではなく任意の救済手段)。

## D-018: ALL CLEAR ボーナス・LEVEL・ランク評価(v3)

- **内容:** 楽しさを増す進行システムを追加した。
  - **ALL CLEAR**: 盤面を空にすると +5000点・ゲージ+50。次の行が600ms以内に降ってくる
  - **LEVEL**(サバイバル): 30秒生存ごとにレベルアップ(+レベル×300点)。行上昇の加速と合わせて進行感を可視化
  - **ランク評価**(サバイバル結果): スコアで S+/S/A/B/C/D を表示
  - HUD に PERFECT ストリーク(🔥×n)を表示
- **理由:** フィードバック「もっとゲームが楽しくなる機能を」。すべて盤面ルールを変えない加点・表示系で、§36 の優先順位(連鎖の気持ちよさ)を損なわない。

## D-019: 特殊ブロックのバランス調整(v4)

- **日付:** 2026-07-20
- **内容:** フィードバック「行上昇が速くなると通常ブロックをほとんど打たず、ボム/プリズムしか打たなくなり、同じ文章を繰り返し打つことになって練習にならない」を受け、2点を修正した。
  1. ボム・プリズムの読み仮名を強制的に short tier(全10件)から選ぶのをやめ、通常ブロックと同じ Tier 分布(short 20% / standard 65% / long 15%、全53件)から選ぶように変更
  2. 盤面全体でボム・プリズムをそれぞれ**同時に最大1個まで**に制限(1行1個の制約は維持)。既存の抽選ロールは、上限に達している種類の分を消費するだけで通常ブロックへフォールバックし、もう片方の出現率を誤って底上げしないようにした
- **理由:** 短文プール(10件)だけを繰り返し引くため文章が単調になり、かつ「短い労力で盤面を大きく消せる」効率の良さから上級者が通常ブロックを無視して特殊ブロックだけを狩る戦略が最適化されてしまっていた。盤面全体で2個(ボム1+プリズム1)までに絞ることで、残り4列以上は通常ブロックの連鎖でしか対処できず、タイピング練習としての多様性が強制的に確保される。

## D-020: ビューポート縮小時のレイアウト保護(v4)

- **内容:** Chromeのブックマークバー表示など、実効ビューポート高さが減った場合に画面上部が到達不能になっていた問題を修正した。原因は `.screen` が `height:100%` 固定+中央寄せで、`body{overflow:hidden}` と組み合わさっていたため、コンテンツがビューポートより高いと中央寄せのはみ出しが上下均等に発生し、上方向のはみ出し分がドキュメント座標マイナスとなってスクロールしても永久に見えなくなっていたこと。
  - `body` を `overflow-y: auto` に変更(下方向のはみ出しは常にスクロールで到達可能にする保険)
  - `.screen` を `justify-content: flex-start` に変更(はみ出しを常に下方向のみへ逃がし、上部が埋もれる経路を構造的に排除)
  - `useFitToViewport` フック(`apps/web/src/hooks/useFitToViewport.ts`)を追加。コンテンツがビューポートに収まらない場合、`transform: scale()` で自動的に縮小する(最小スケール0.55、それ以上縮めても読みにくくなるため下限を設け、それでも収まらない極小ウィンドウはスクロール併用にフォールバック)。ResizeObserver(コンテンツ自身+documentElementの双方を監視)・window resizeイベント・500msポーリングの3系統で検知し、いずれか1つが効かない環境でも追従する
- **理由:** ブックマークバーの表示/非表示はページのコンテンツサイズを一切変えずビューポートだけを変えるため、コンテンツ側だけを監視していると検知できない。検証で使用したブラウザツールの `resize_window` はCDP経由のビューポート上書きで、実ブラウザと異なり `resize` イベントも `ResizeObserver` も発火しないことが判明したため、ポーリングを保険として追加した(実ブラウザでは通常イベントベースの検知だけで十分機能する)。
