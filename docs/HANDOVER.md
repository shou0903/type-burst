# HANDOVER — 新しく入るAI/開発者への引継書

**このプロジェクトで作業する前に、まずこの文書を最初から最後まで読むこと。**
このリポジトリは既に60件近い決定(`docs/DECISIONS.md`)を経ており、同じ議論や同じ失敗を
繰り返さないためにこの文書がある。個別の実装詳細は他のdocsやコードそのものにあるので、
ここでは「読む前に必ず知っておくべきこと」だけに絞る。

## 0. 一分でわかる概要

- **これは何か:** 日本語の文章・単語をローマ字で入力してブロックを爆破し、同属性の落下連鎖を
  起こすブラウザパズルゲーム「TYPE BURST」。
- **公開URL:** https://type-burst-web.vercel.app/
- **リポジトリ:** https://github.com/shou0903/type-burst(`master`ブランチへのpushでVercelが自動デプロイ)
- **正本仕様書:** [`TYPE_BURST_Product_Game_Technical_Spec_v1.0.md`](../TYPE_BURST_Product_Game_Technical_Spec_v1.0.md)
  だが、**この仕様書は初期構想であり、以後の全ての変更・上書きは`docs/DECISIONS.md`に記録されている。**
  仕様書とDECISIONS.mdが矛盾する場合は**DECISIONS.mdが常に正**(新しい方が優先、末尾が最新)。
- **利用者はコーディングを他のAIにも依頼する運用**なので、この文書はAIが変わっても
  一貫した挙動・品質を保つための引継書として書かれている。

## 1. 絶対に破ってはいけないルール

過去に何度も同じ議論が発生し、そのたびにユーザーが同じ結論へ戻している事項。
**「良かれと思って」判断だけで変更しない。** 変更が必要に見える場合は、まずこのルールが
既にユーザーとの合意事項でないかをDECISIONS.mdで確認してから提案すること。

1. **サバイバルの行上昇速度(rise interval)は全難易度で完全に共通。**
   難易度差は「1ブロックあたりに要求される文章の長さの配分」(`tierRatio`)だけで表現する。
   行上昇速度を難易度ごとに変える案は**D-032→D-038で2度試して2度とも撤回**されている
   (D-033, D-038, D-039, D-041, D-059を参照)。この制約を破る変更は、たとえ別のバグ修正が
   目的でも行わないこと。
2. **キャラクター・ストーリー・スキル・課金要素(ガチャ等)を追加しない。**
   盤面ギミック(ボム・プリズム・TYPE BURST)の範囲に留める(D-013)。
3. **外部AI APIを使わない。**
4. **標準問題は自然な日本語のみ。** 英字だけの文字列・URL・メールアドレス・ファイル名・
   コード断片、歌詞/小説/台詞の引用、差別・性的表現、政治宗教勧誘、医療/投資助言、
   商品広告は禁止(詳細は`docs/CONTENT_GUIDELINES.md`)。
5. **`packages/game-core`はSeed+呼び出し列に対して完全に決定論的。**
   `Math.random()`は使わず、wall-clockにも依存しない。「演出のためにシミュレーション時間を
   歪める」ような変更(例: 本物のスローモーション)は決定論テストを壊すため避ける
   (D-051で実際にこの理由から設計を変えた実例あり)。
6. **BGM(ループ音源)は追加しない。** 一度追加してユーザーの意向で即日撤去された経緯がある
   (D-036)。単発SFX+画面演出は問題ない。
7. **モバイル最適化(スマホで実際にプレイできるようにする)はスコープ外のまま。**
   物理キーボード前提を変えない。TikTok等モバイル流入への対応は「専用ランディングページで
   PCへの再訪を促す」方針(D-057)であり、「スマホでも遊べるようにする」方向には進めない。
8. **広告(Google AdSense)は表示するが、Pay-to-Winは禁止のまま。**
   仕様書の「広告なし」方針からの明示的な転換(D-027)だが、ゲーム進行・結果への影響は禁止。
9. **ユーザー登録は不要のまま。** ランキング参加用のニックネームのみ任意入力。

## 3. アーキテクチャ地図

```text
apps/web ──────────────┐
                       ▼
        packages/game-core ──▶ packages/typing-engine
                       │                ▲
                       ├──▶ packages/phrase-content
                       │
apps/web ─────────▶ packages/progression
```

- 依存は常に上から下へ。`typing-engine`は何にも依存しない。
- 各パッケージはTypeScriptソースを直接公開し、Vite/Vitestがエイリアス解決する(ビルドステップなし)。
- **`packages/progression`(新しめ)**: 称号ラダー(`titles.ts`)と生涯累計統計の集計(`progress.ts`)
  を持つ。`apps/web`の`vitest.config.ts`(後述の理由で編集不可な時期があった)を変更せずに
  単体テストを書くために切り出された経緯がある。**盤面カラーテーマ(`boardThemes.ts`)は
  ここに同居していたがD-059で削除済み**なので、もし過去のコミット履歴で見かけても復活させない。

### パッケージ別の要点

- **`packages/typing-engine`**: かな→ローマ字受理グラフ(NFA)。shi/si・chi/ti・促音・ん・
  拗音の複数表記を受理する。
- **`packages/game-core`**: DOM非依存の純ロジック。`PlayerCore`が単一プレイヤーの盤面を
  駆動し、`SurvivalGame`/`DuelGame`/`TutorialGame`がそれぞれのモード固有ルールを被せる。
  `advance(deltaMs)`と`feedKey(key)`だけで駆動され`GameEvent[]`を返す。
- **`packages/phrase-content`**: 出題データ(`phrases.ts`)と検証(`validatePhrases()`)。
  tier(micro/short/standard/long)ごとにモーラ帯域が決まっている。
- **`apps/web`**: React + Canvas 2D + Web Audio。`GameController`がrAFループで
  `advance`を呼び、`BoardRenderer`(Canvas描画)と`SoundEngine`(Web Audio合成)へ配る。
  盤面計算をReactコンポーネントに書かない。
- **`apps/x-automation`・`apps/web/api/social/`**: ゲーム本体とは別の施策(SNS自動投稿系)。
  詳しくは第5章を参照。

### 開発時デバッグフック

`import.meta.env.DEV`時のみ、`window.__typeblastDebug`にその時点の`GameController`
インスタンスが載る。ブラウザでの動作検証はほぼ全てこれを使う:

- `debugAdvance(ms)`: 時間を進める(内部で50ms刻みにスライスして`advance`を呼ぶ)
- `debugType(keys)`: 文字列を1文字ずつ`feedKey`する
- `debugSnapshot()`: 現在のスナップショットを取得(`blocks`・`toppedOut`等)
- `debugRomajiFor(displayText)`: 盤面上の指定フレーズの正しいローマ字を取得
- `debugBurst()`: TYPE BURSTを発動

本番ビルドには含まれない。

## 4. 開発ワークフロー(必須)

### 検証パイプライン

コミット前に必ず全て通す:

```bash
npm run typecheck && npm test && npm run build
```

### Git運用ルール

- **`git add -A` / `git add .` は絶対に使わない。** 必ず変更したファイルを明示的に指定して
  `git add`する(理由は第5章)。
- コミット前に`git status --short`と`git diff --cached --stat`で、意図したファイルだけが
  staged になっているか必ず確認する。
- 新しいコミットを作るのが基本方針(`--amend`は明示的に頼まれない限り使わない)。
- コミットメッセージは日本語で、末尾に`Co-Authored-By:`行を付ける(既存コミット履歴を参照)。

### `docs/DECISIONS.md`への記録

- 仕様からの逸脱・設計判断・ユーザーからの明確な指示による変更は、**必ず末尾に追記**する。
- フォーマット: `## D-0XX: タイトル` → `- **日付:**` → `- **内容:**` → `- **理由:**` →
  (再発防止に値する学びがあれば)`- **教訓:**`。
- 追記前に最後のD番号を確認し、連番でずれないようにする。
- ユーザーが一度決めた方針を後から自分の判断だけで覆さない(D-038→D-039のように、
  過去に一度この失敗をして訂正されている)。

### デプロイ確認

`master`へpushすると自動デプロイされる。確認は本番のJSバンドルハッシュが変わったかを
ポーリングする:

```bash
curl -s https://type-burst-web.vercel.app/ | grep -o 'assets/index-[A-Za-z0-9_-]*\.js'
```

ハッシュが変わったら、実際に本番URLへブラウザでアクセスしてコンソールエラーがないこと、
主要な導線(ランディング→サバイバル開始→分析画面等、変更に関係する画面)が動くことを
確認してから完了報告する。

## 5. ⚠️ 並行セッションの危険信号(このリポジトリ特有・時限的な注意)

**このドキュメント作成時点で、別のAIセッションが同じ作業ディレクトリで並行して
別タスク(SNS自動投稿・マーケティング施策)に取り組んでいる。** 以下は現時点で
そのセッションの管理下にあり、**絶対に編集・コミットしないこと**:

- `apps/x-automation/`
- `apps/web/api/social/`
- 直下の`package.json` / `tsconfig.json` / `vitest.config.ts`
- `docs/SNS_MARKETING_BRIEF.md` / `docs/TIKTOK_GEMINI_PROMPT.md`
- `marketing-drafts.md`

**この一覧は時間とともに変わる。** 上記のタスクが完了してコミットされれば、この注意書き
自体が古くなる。次にこの作業をするAIは、**必ずセッション開始時に`git status --short`を
実行し、見覚えのない未コミットの変更・未追跡ファイルがあれば「誰かの作業中の変更かもしれない」
と考えて、触る前に調査する**こと(削除・上書きしない。必要ならユーザーに確認する)。

このリポジトリはゲーム本体(`packages/*`, `apps/web`のゲームUI)とは無関係な複数の
施策が同時並行で走ることがある前提で運用されている。自分のタスクに関係ないファイルは
たとえ`git status`に出ていても触らない、という原則を徹底すること。

## 6. この環境(サンドボックス)特有の制約

ブラウザでの動作検証を行う際、以下の既知の制約がある(実際にこれらが原因で
検証漏れ→本番バグが起きた実例がD-058にある):

- **`document.hidden`が常に`true`で`requestAnimationFrame`が発火しない。**
  RAFに依存するアニメーションループ(例: `AttractBoard.tsx`)は、この環境では
  一度も実行されないまま「動いているように見える」ことがある。検証には
  `debugAdvance()`等による手動フレーム送り、またはロジックを直接シミュレートする
  スクリプトを使うこと。「コンソールエラーが出ない」ことは「動いている」ことの
  証明にならない。
- **スクリーンショット撮影(`computer`ツールの`screenshot`/`zoom`)がタイムアウトして
  機能しないことがある。** その場合は代替手段で検証する:
  - `javascript_tool`で`getBoundingClientRect()`を使い、要素の実座標・サイズを直接検証
  - `get_page_text`でテキスト内容を検証
  - `read_page`でDOM構造・アクセシビリティツリーを検証
- **`navigator.clipboard`への書き込みが常に拒否される。** クリップボードコピー機能は
  失敗時のフォールバック経路しかこの環境では検証できない。
- 最終成果物(ピクセル・DOM座標・テキスト)を直接調べる方法が、実行経路をトレースできない
  環境でも唯一信頼できる検証手段になることが多い。

## 7. サブエージェント(他AI)へ委譲する場合の注意

- **独立性の高い複数機能は、同じファイルを触らない単位に分けて順番に実行する。**
  並行で走らせるとファイル競合・上書きが起きる。
- **各タスク完了後は自分で実際の成果物を検証してから次へ進む。** サブエージェントの
  「完了しました」という自己申告をそのまま信用しない。実際に:
  - 本当にコードが変わっているか(`git diff`)
  - テスト・typecheck・buildが通るか
  - ブラウザで実際に動くか(第6章の制約を踏まえた方法で)
  を確認する。D-058では、サブエージェントが「動作確認済み」と報告した機能が、
  実際には本番の主要デバイス層(60Hz以上の端末)で完全に描画されていない致命的バグを
  含んでいたことが、独立した検証(canvasの直接ピクセルサンプリング)によって発覚した。
- セッション制限等でサブエージェントが途中で止まった場合、最初からやり直すのではなく、
  `git log` / `git status` / 差分を読んで「どこまで終わっているか」を把握してから
  続きを引き継ぐ方が速い。

## 8. 過去の失敗パターン(繰り返さないこと)

`docs/DECISIONS.md`から特に再発しやすいものを抜粋:

1. **「終了条件に効くパラメータ」と「体感の良さに効くパラメータ」を混同する。**
   難易度調整で「行上昇の速さ(終了条件の上限)」と「文章の長さ(体感難易度)」を混ぜて
   同時に変えると、意図せず「無限に生存できる」バグを作り込みやすい(D-032, D-041, D-059)。
2. **ユーザーが明示的に確定した設計制約を、別の不具合修正のために自分の判断だけで緩める。**
   D-038でこれをやってしまい、即日ユーザーから訂正された。制約変更が必要に見えたら、
   まず「これは以前ユーザーが除外した選択肢ではないか」をDECISIONS.mdで確認する。
3. **「短くして」「もっと増やして」等の曖昧な要求を、既存カテゴリの延長線上でしか解釈しない。**
   D-040では「短文」を要求されて既存の文章tierを縮めようとしたが、ユーザーの意図は
   「単語単体(寿司打のような)」という別カテゴリだった。早い段階で具体例を確認すべきだった。
4. **機能の「一部だけ撤去」の意図を取り違える。**
   D-059(このセッションの直前タスク)では「語彙テーマは要らないが単語自体は残す」という
   指示があった。UIとロジックだけを撤去し、コンテンツ(語彙データ)は絶対に消さないこと。
   同様に、称号システムと盤面カラーテーマは同じ`packages/progression`パッケージに同居して
   いたが無関係の機能であり、後者だけを正確に切り離して削除する必要があった。
5. **検証環境の制約(第6章)を「未検証」ではなく「検証済み」と報告してしまう。**
   RAFが動かない・スクリーンショットが撮れない、といった制約下で通しただけのテストを
   「実機で確認済み」のように報告しない。何を、どの方法で、どこまで検証できたかを正直に書く。
